# 功能規格：使用者歷程時間軸

**功能分支**：`006-user-journey-timeline`  
**建立日期**：2026-06-22  
**狀態**：草稿  
**輸入**：使用者描述：「在『我的職場屬性報告』及管理員編輯使用者時的『職場屬性報告』Tab 內加入使用者歷程時間軸，由近到遠顯示非餐費補助申請與管理員建立的事件；事件包含標題、內容、附件等新資料。」

> 本文件只記錄產品需求、使用者價值、驗收條件與成功標準；技術實作記錄於 plan.md。

## 文件範圍確認

- 「事件」是本功能新增的業務資料，不等同既有申請 audit trail 或考核資料。
- 時間軸只彙整目標使用者自己的非餐費補助申請與事件，不包含出勤、餐費補助或系統稽核紀錄。
- 所有已登入使用者均可跨使用者讀取事件與其附件；事件的新增、更新與刪除僅限 Admin 操作，目標使用者與非目標使用者皆只能讀取。本功能的 Admin 專指目前登入者在 `users/{request.auth.uid}` 文件中的 `role` 欄位等於 `admin`。

## 使用者情境與測試

### 使用者故事 1 - 查看個人歷程 (Priority: P1)

使用者在「我的職場屬性報告」頁面查看自己的歷程，時間軸將非餐費補助申請與事件合併，依業務發生時間由近到遠呈現。

**獨立驗證方式**：為同一使用者建立不同日期的補助與事件，確認時間軸跨類型正確排序，且不出現餐費或他人資料。

**驗收情境**：

1. **Given** 使用者有多筆補助申請與事件，**When** 開啟「我的職場屬性報告」，**Then** 報告內容後方顯示時間軸，所有項目由近到遠排列。
2. **Given** 使用者有筆電、健檢、訓練、AI 工具或旅遊補助，**When** 查看時間軸，**Then** 每筆顯示申請類型、申請日期、狀態、金額與可用的內容或附件。
3. **Given** 系統存在餐費補助或其他使用者資料，**When** 查看自己的時間軸，**Then** 這些資料不會出現。
4. **Given** 使用者沒有任何歷程，**When** 開啟頁面，**Then** 顯示清楚的空狀態，職場屬性報告本身仍可正常使用。

---

### 使用者故事 2 - 建立與維護事件 (Priority: P1)

管理員在編輯指定使用者時，可於「職場屬性報告」Tab 的時間軸建立、編輯及刪除事件。目標使用者與非目標使用者皆可讀取事件與附件，但不能新增、更新或刪除事件。事件至少包含發生日期、事件標題與事件內容，並可選擇附加檔案。

**獨立驗證方式**：以管理員建立、修改與刪除事件；再以目標使用者與非目標使用者確認可讀但不可寫。

**驗收情境**：

1. **Given** 管理員正在編輯使用者，**When** 輸入發生日期、標題、內容及合法附件並儲存，**Then** 新事件立即出現在該使用者時間軸的正確位置。
2. **Given** 既有事件，**When** 管理員修改內容、日期或附件，**Then** 時間軸顯示最新內容並依更新後的發生日期重新排序。
3. **Given** 既有事件，**When** 管理員確認刪除，**Then** 事件與其附件不再可見，但保留足以追查操作者與時間的異動紀錄。
4. **Given** 目標使用者查看自己的時間軸，**When** 讀取事件或附件，**Then** 系統允許讀取且不提供新增、編輯或刪除事件操作。
5. **Given** 已登入的非目標使用者查看事件，**When** 讀取事件或附件，**Then** 系統允許讀取；嘗試建立、更新或刪除時則拒絕。

---

### 使用者故事 3 - 查看歷程附件與更多資料 (Priority: P2)

所有已登入使用者可查看時間軸項目的附件；歷程很多時可逐批載入更早資料，不必一次下載全部紀錄。

**獨立驗證方式**：準備超過一頁的混合資料與圖片、PDF 附件，確認預覽、分頁、排序與失敗提示。

**驗收情境**：

1. **Given** 時間軸項目包含圖片或 PDF，**When** 點選附件，**Then** 系統沿用申請附件的一致預覽體驗。
2. **Given** 歷程超過首批顯示數量，**When** 選擇「載入更早歷程」，**Then** 新資料接續在下方且整體排序不重複、不遺漏。
3. **Given** 任一資料來源或附件暫時載入失敗，**When** 頁面顯示結果，**Then** 系統提供繁體中文錯誤與重試操作，不將失敗誤顯示為無資料。

### 邊界案例

- 同一時間有多筆不同類型資料時，以穩定次序顯示，重新載入不得跳動。
- 事件日期可早於建立日期；排序必須使用事件發生日期，而非資料建立日期。
- 補助排序使用既有申請日期；狀態或內容更新不改變其時間軸位置。
- 事件標題或內容過長時需安全換行，附件名稱含特殊字元或不同語系時仍可正常顯示。
- 使用者在瀏覽期間離職、事件被刪除或權限改變時，重新讀取與儲存仍以當下權限為準。
- 刪除事件或替換附件失敗時，不得留下無資料關聯的附件；舊資料不得因部分失敗而損壞。

## 需求

### 功能需求

- **FR-001**：系統 MUST 在「我的職場屬性報告」頁面的報告內容後方顯示登入使用者的歷程時間軸。
- **FR-002**：系統 MUST 在管理員編輯使用者時的「職場屬性報告」Tab 內，於報告內容後方顯示該使用者的歷程時間軸。
- **FR-003**：時間軸 MUST 合併目標使用者的非餐費補助申請與事件，並依業務發生時間由近到遠排序。
- **FR-004**：時間軸 MUST NOT 顯示餐費補助、其他使用者資料、出勤紀錄或 audit trail。
- **FR-005**：補助項目 MUST 至少顯示補助類型、申請日期與申請狀態；有金額、內容及附件時 MUST 提供相應摘要或檢視能力。
- **FR-006**：系統 MUST 新增事件資料定義，至少包含目標使用者、事件發生日期、事件標題、事件內容、零至五個附件、建立者、建立時間、最後修改者與最後修改時間。
- **FR-007**：事件標題 MUST 為必填且最多 100 字；事件內容 MUST 為必填且最多 5,000 字。
- **FR-008**：所有已登入使用者 MUST 能跨使用者讀取所有事件；只有 Admin MUST 能建立、更新及刪除任意事件；目標使用者與非目標使用者 MUST NOT 能寫入事件。Admin MUST 由目前登入者對應的 `users/{userId}.role == "admin"` 判定。
- **FR-009**：Admin MUST 能在既有使用者編輯流程直接建立、更新及刪除該使用者事件；「我的職場屬性報告」中的事件 MUST 僅提供讀取，不提供新增、更新或刪除操作。
- **FR-010**：事件附件 MUST 完整沿用 attendance/subsidy 申請附件的規章，包括每事件最多五檔、每檔不超過 3 MiB、格式限 PDF／JPEG／PNG／WebP，以及相同的驗證、預覽、替換、失敗補償、清理與孤兒檔治理行為。
- **FR-011**：系統 MUST 允許所有已登入使用者讀取及預覽任意事件的圖片與 PDF 附件；未登入者 MUST NOT 能存取。
- **FR-012**：事件與附件的新增、修改及刪除 MUST 避免部分完成，並保留建立、修改、刪除的操作者及時間供查核。
- **FR-013**：時間軸 MUST 分批載入，首批及後續每批最多顯示 20 筆合併結果。
- **FR-014**：相同時間的項目 MUST 以穩定規則排序：事件優先於補助，再以文件識別碼排序。
- **FR-015**：資料或附件載入失敗時，系統 MUST 顯示繁體中文錯誤及重試操作，不得將錯誤狀態顯示為空狀態。

### 主要實體

- **時間軸項目**：畫面層統一表示補助申請或事件，具有來源類型、發生時間、標題、摘要、附件與穩定排序鍵。
- **事件**：Admin 為單一目標使用者建立的職場歷程紀錄，具有業務發生日期、標題、內容、附件與建立／修改資訊。
- **補助申請**：既有非餐費補助申請；本功能唯讀呈現，不改變其申請生命週期。
- **事件異動紀錄**：事件建立、修改或刪除的稽核資料，保留實際操作者、操作時間與必要摘要。

## 假設與範圍

- 「我的申請」指 `subsidyApplications` 中 `userId` 為目標使用者的資料；餐費資料位於其他集合，因此不納入。
- 首版不提供全文搜尋、日期篩選、事件分類、留言、通知或匯出。
- 事件刪除需二次確認；刪除後不在時間軸顯示，稽核紀錄仍保留。
- 事件附件採既有申請附件相同限制，以維持一致體驗；若產品日後需要影片或更大檔案，另案調整。

## 成功標準

- **SC-001**：包含 100 筆混合歷程時，95% 的首批時間軸在一般網路下於 3 秒內顯示內容或明確載入狀態。
- **SC-002**：混合排序、同時點穩定排序、分頁、餐費排除與使用者隔離的自動化測試通過率為 100%。
- **SC-003**：Admin 可在 2 分鐘內完成一筆含附件事件的建立，且目標使用者重新開啟報告後可看到正確資料。
- **SC-004**：所有已登入使用者跨使用者讀取事件／附件的允許率為 100%；非 Admin 寫入事件及未登入讀取事件／附件的拒絕率為 100%。
- **SC-005**：模擬事件建立、附件替換與刪除任一步驟失敗後，部分完成資料與真正孤兒附件數量皆為零。
