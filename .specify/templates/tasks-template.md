# 任務清單：[功能名稱]

**輸入**：來自 `/specs/[###-feature-name]/` 的設計文件  
**前置條件**：plan.md（必要）、spec.md（必要）、research.md、data-model.md、contracts/

**測試要求**：凡涉及 business logic、權限判斷、Firebase 資料存取或跨模組流程者，MUST 同時包含單元測試與整合測試任務；測試不是可選項。

**組織原則**：任務必須依使用者故事分組，使每個故事都能獨立實作、獨立測試、獨立展示價值。

## 格式：`[ID] [P?] [Story] 說明`

- **[P]**：可平行執行（不同檔案、無相依）
- **[Story]**：任務所屬使用者故事（例如 US1、US2、US3）
- 每項任務說明 MUST 包含精確檔案路徑

## 路徑慣例

- Angular 應用程式：`src/app/`、`src/environments/`、`public/`
- Firebase 規則與索引：`firestore.rules`、`firestore.indexes.json`
- 工具腳本：`tools/`
- 測試檔案：優先沿用功能檔相鄰的 `*.spec.ts`，必要時補充整合測試目錄或模擬器測試腳本

## 相依性與執行順序

### Phase 相依性

- **Phase 1：準備**：無相依，可立即開始
- **Phase 2：基礎能力**：依賴 Phase 1 完成；會阻塞所有使用者故事
- **Phase 3 之後的使用者故事**：依賴 Phase 2 完成；之後可依資源平行進行
- **最終階段**：依賴所有欲交付的使用者故事完成

### 使用者故事相依性

- **US1 (P1)**：完成基礎能力後即可開始，作為 MVP 優先交付
- **US2 (P2)**：完成基礎能力後可開始，但不得破壞 US1 的獨立可測性
- **US3 (P3)**：完成基礎能力後可開始，若依賴前述故事須明確標示

### 每個故事內部順序

- 測試 MUST 先寫並先失敗，再開始實作
- 資料模型／型別先於服務
- 服務先於元件整合
- Security Rules、索引與文件更新不得晚於功能完成後補寫
- 單一故事完成後必須先驗證其可獨立展示與測試，再進入下一優先級

### 可平行工作

- 所有標記 `[P]` 的任務可平行執行
- 不同使用者故事可在基礎能力完成後由不同成員平行處理
- 測試、資料模型與部分 UI 任務可分工進行，但同一檔案不可平行修改

## 實作策略

### MVP 優先

1. 完成 Phase 1 與 Phase 2
2. 完成 US1 的測試與實作
3. 驗證 US1 的功能、授權、成本與文件
4. 視需要先行示範或部署 MVP

### 漸進式交付

1. Foundation 完成後，優先交付 US1
2. 每新增一個故事，都要重新驗證既有故事不回歸
3. 每次交付都必須同步更新測試、Security Rules 與繁體中文文件

## 備註

- 任務描述不得含糊，必須指向具體檔案與可驗證成果
- 不可將 Firebase-only、測試或 Security Rules 工作留到「之後再補」
- 若發現現有遺留設定（例如 skipTests 或 Cloudinary 腳本）與本功能衝突，必須在任務中明確處理
